Home Page — Configuration & Workspaces

The home page is the launchpad for Prompt Constructor. Before you build or run anything, this is where you confirm that the local services the app depends on are actually reachable.

Connection Configuration has three model roles: LLM for chat generation, Embed for semantic memory search, and Classifier for detecting tags or scenarios that trigger memory rules.

Connection Status checks whether the browser can reach each local service. If a check fails, that part of the app needs its URL, model name, server process, or CORS setting fixed.

Workspaces are how user data is separated without requiring login or auth. The workspace ID is tied to saved registries, connection settings, snapshots, memory files, personality notes, working notes, summaries, and similar artifacts. If that workspace is cleared from browser storage, or you open the site from a different computer, the old workspace data will not be available unless you still have that workspace ID and its saved files. To make sure you never lose model registries you build, export and store the registry JSON; it can be imported back into the site as raw JSON from any workspace.

1. The Problem

If you want an LLM's behavior to vary by persona, mood, context, or conversation state, hardcoded prompt strings stop scaling. As soon as you want strict control over dynamic behaviorial presedence, and clear auditing capabilities, is when you need to adopt and adapt to manage that complexity. Promptlibretto is an effective abstraction layer and is not opinionated about how you structure your prompts internally. Promptlibretto is a simple organizational framework that has clear mechanics for how to manage a wide range of use cases while keeping the system auditable and maintainable. Typical patterns that emerge without a systematic approach are:

  • Hard to navigate as a result of concatenated strings scattered across the codebase
  • inconsistent prompt construction leading to irreproducible results
  • no reliable audit trail to track changes
  • no clear understanding of what the model is actually receiving

PromptLibretto as a sysytem is designed to ORchestreates this kind of complex behavior. through layed registries, we can apply distinct behavior rules sequentionally, enabling realtime adaptation when registry definition or input context changes dynamically. The framework builds the prompt structure before finiling it with runtime data.

5. Builder

The Builder is a visual authoring tool for registry JSON. Instead of writing large JSON by hand, you author sections, items, fragments, and groups through a structured UI.

The assembly order panel shows exactly which tokens will appear in the final prompt and in what sequence. Section cards show each bucket of authored content. Item detail shows context text, conditional fragments gated by template variables, and behavioral directive groups attached to each item.

Registries can be exported as JSON, imported back, or sent directly to Studio for testing.

Intro — Prompt Libretto & Prompt Constructor

The registry is a JSON document that defines every section of a prompt, the relationships between them, and the assembly order that controls what the model actually receives. At generation time, the engine walks that order, resolves the current state, fills template variables, and produces the final prompt as a single string. That is the core mechanic — a declared, auditable assembly path instead of scattered string concatenation.

The problem it replaces is this: as soon as you want a language model's behavior to vary by persona, mood, context, or conversation state, hardcoding prompt strings stops working. You end up with concatenated strings scattered across the codebase, no consistency, and no way to audit what the model is actually receiving. Prompt Libretto is the library that replaces that with a registry-based architecture.

The main building blocks are Registry, Section, RegistryState and SectionState, and a set of item helpers — Persona, Sentiment, Group, StaticInjection, RuntimeInjection, OutputDirection, and PromptEnding. The Registry is the complete blueprint, loaded once and read-only at runtime. Sections are named buckets of authored items. RegistryState and SectionState hold per-call choices — which item is selected, what the slider value is, what template variables are filled in. The engine's hydrate method resolves all of that into the final prompt string.

Prompt Constructor is the studio built on top of that library. It has four surfaces: the Builder for visually authoring registry JSON, the Studio for loading a registry, tuning runtime state, previewing the assembled prompt, and generating against a live model, the Chat Builder for building registries through conversation, and Ensemble for running two registry-driven agents in one autonomous conversation.

The memory system gives agents continuity without replaying the entire conversation every turn. Each exchange is embedded into a vector and stored locally. Before the next response, relevant prior exchanges are retrieved and injected as memory_recall. A classifier can also inspect incoming messages, generate tags, and fire rules that inject temporary guidance or modify emotional state variables like warmth, trust, tension, and playfulness. Those states accumulate and decay over time, and style blending lets them shift behavior gradually without replacing the base persona. Working notes give the model a lightweight scratchpad it updates every few turns to maintain orientation over long interactions.

Ensemble is where it gets more interesting. Each participant has memory, summarization, emotional state, working notes, and retrieval. The scenario is steered by domain, persona, tone, and behavioral constraints, but the output is non-deterministic. Think of it as the difference between scripted dialogue and improv with rules. The registry sets the stage and the character. What actually gets said is up to the model.

The Chat Builder is a two-pane interface — assistant chat on the left, live registry visualization on the right. You describe what you want and it streams tool calls that build the registry incrementally. Unlike Studio and Ensemble, the browser does not call the model directly here. Requests go to the FastAPI server at /api/builder/chat, which calls the model server-side, so no CORS configuration is needed.

For Studio and Ensemble, the browser does talk directly to your local model server, so that server must allow the browser origin through CORS. The privacy model is simple: prompt assembly happens on the Prompt Libretto server, the assembled prompt is returned to the browser, and the browser sends it directly to your local model. Prompt Libretto is not an LLM proxy. Prompts, responses, embeddings, and classifications stay on your machine.

Builder Walkthrough — Wizard Tech Support

So this is the Wizard Tech Support registry loaded in Builder.

At the top we have the assembly order — this is the backbone of the system. It defines exactly what sections get included in the final prompt, and in what order. Everything flows from here.

Base context sets the scene — this agent works for a magical helpdesk. There's a template variable for the company name, so you can swap that out per deployment.

Personas give the prank caller a mask. There are two here: the old_lady persona, which mishears things and wanders off topic, and my_dad_owns_this_company, which is the selected persona. That selected persona also has a behavior group, and its array mode picks two of those group items at random so the call can vary without rewriting the persona itself.

Sentiment is separate from persona. This registry has a sarcastic option and a selected core_personality option. Core personality has a superiority-complex scale with slider_random turned on, plus a traits group where two traits are randomly selected at runtime.

Static injections are fixed prompt blocks. This registry does use one: phrases. It adds the "my dad owns the company" flex and can randomly pull two put-down lines from its group.

Runtime injections are dynamic request-time blocks. This registry has the section available, but it is empty for this prank caller.

Memory rules are where it gets interesting — if the classifier detects the agent is being asked for an order number, product ID, or purchase code, it appends a private instruction. The prankster should stall for a second, pretend to look for the number, and then make one up with total confidence.

Groups are where reusable behavioral snippets live. For example, a group like when_responding can provide a pre-context line and a list of response behaviors, then the selected persona or sentiment can attach that group without duplicating the same instructions in multiple places.

And at the bottom, output directions and prompt endings cap it off — format rules and the role label that kicks off the model's response.
