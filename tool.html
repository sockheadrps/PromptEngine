Home Page — Configuration & Workspaces

The home page is the launchpad for Prompt Constructor. Before you build or run anything, this is where you confirm that the local services the app depends on are actually reachable.

Connection Configuration has three model roles: LLM for chat generation, Embed for semantic memory search, and Classifier for detecting tags or scenarios that trigger memory rules.

Connection Status checks whether the browser can reach each local service. If a check fails, that part of the app needs its URL, model name, server process, or CORS setting fixed.

Workspaces are how user data is separated without requiring login or auth. The workspace ID is tied to saved registries, connection settings, snapshots, memory files, personality notes, working notes, summaries, and similar artifacts. If that workspace is cleared from browser storage, or you open the site from a different computer, the old workspace data will not be available unless you still have that workspace ID and its saved files. To make sure you never lose model registries you build, export and store the registry JSON; it can be imported back into the site as raw JSON from any workspace.

Intro — Prompt Libretto & Prompt Constructor

Prompt Libretto is a Python library for building structured, dynamic system prompts — at runtime, from a declarative model called a registry.

The problem it solves is this: as soon as you want a language model's behavior to vary — by persona, mood, context, or conversation state — hardcoding prompt strings stops working. You end up with concatenated strings scattered across your codebase, no consistency, and no way to audit what the model is actually receiving. Prompt Libretto replaces that with a registry: a JSON document that defines every section of a prompt, how sections relate to each other, what's required, what's optional, and what gets selected or randomized at call time. The engine assembles the final prompt from that spec every time.

This tool — Prompt Constructor — is one example of how you can build on top of that. It's a studio for creating chat models: you get a visual builder for authoring registries, a studio for testing them against a live model, and an ensemble mode that runs two registry-driven agents in conversation with each other.

One important setup detail: because the browser talks directly to your local model server, that model server has to allow this browser origin with CORS. For Ollama on Windows, you can start it with OLLAMA_ORIGINS set to http://127.0.0.1:8181, then run ollama serve. Without that, the browser blocks the request before it ever reaches the model, so the server may look quiet even though generation is failing.

The privacy model is simple: browser to your local model, and back. The promptlibretto server running at http://127.0.0.1:8181 is not the LLM proxy. It assembles the registry into prompt text and returns that prompt to the browser. Then the browser sends the prompt to your local generation, classification, or embedding model. Your prompts and model responses stay on your machine and do not travel through the promptlibretto server.

With memory enabled, the server may coordinate the memory index, but model calls still happen from the browser. The server asks the browser for embeddings over a lightweight WebSocket; the browser sends text to your local embedding model, gets vectors back, and returns those vectors to the server index. The server stores the searchable vector data, not the raw text sent to the local model.

Ensemble is where it gets more interesting. Each participant has memory — an embedding model indexes the conversation as it happens, a summarizer compresses older context into a rolling system summary, and working notes let each agent track its own read on the situation. The scenario is steered — you define the domain, the persona, the tone — but the output is non-deterministic. The model generates freely within those constraints, so each run produces something unique while still staying true to the theme.

Think of it as the difference between scripted dialogue and improv with rules. The registry sets the stage and the character. What actually gets said is up to the model.

Builder Walkthrough — Wizard Tech Support

So this is the Wizard Tech Support registry loaded in Builder.

At the top we have the assembly order — this is the backbone of the system. It defines exactly what sections get included in the final prompt, and in what order. Everything flows from here.

Base context sets the scene — this agent works for a magical helpdesk. There's a template variable for the company name, so you can swap that out per deployment.

Personas give the prank caller a mask. There are two here: the old_lady persona, which mishears things and wanders off topic, and my_dad_owns_this_company, which is the selected persona. That selected persona also has a behavior group, and its array mode picks two of those group items at random so the call can vary without rewriting the persona itself.

Sentiment is separate from persona. This registry has a sarcastic option and a selected core_personality option. Core personality has a superiority-complex scale with slider_random turned on, plus a traits group where two traits are randomly selected at runtime.

Static injections are fixed prompt blocks. This registry does use one: phrases. It adds the "my dad owns the company" flex and can randomly pull two put-down lines from its group.

Runtime injections are dynamic request-time blocks. This registry has the section available, but it is empty for this prank caller.

Memory rules are where it gets interesting — if the classifier detects the agent is being asked for an order number, product ID, or purchase code, it appends a private instruction. The prankster should stall for a second, pretend to look for the number, and then make one up with total confidence.

Groups are where reusable behavioral snippets live. For example, a group like when_responding can provide a pre-context line and a list of response behaviors, then the selected persona or sentiment can attach that group without duplicating the same instructions in multiple places.

And at the bottom, output directions and prompt endings cap it off — format rules and the role label that kicks off the model's response.

Studio Walkthrough — Wizard Tech Support

Now here's the same registry running in Studio — this is where you actually talk to it.

On the left you can see the live state panel. Every section from the assembly order shows up here — persona, sentiment, static injections — and you can change any of them without touching the JSON. Switch between old_lady and my_dad_owns_this_company, let the sentiment scale randomize, toggle the phrases injection, and the assembled prompt updates instantly.

Down here is the prompt preview. This is exactly what gets sent to the model — no guesswork. You can see the persona context, the behavioral directive that was randomly picked this run, the sentiment nudge, and the static injection text all assembled in order.

I'll type something in as the support agent. Notice the model responds in character as the caller — entitled, weirdly confident, and trying to turn a fake support issue into an escalation.

Now watch what happens when I switch personas to old_lady and ask the same kind of question. Same registry, same output rules, completely different voice. The prompt preview updates and the model responds accordingly.

The output policy is enforced silently in the background. This one sets a minimum length of 80 and a maximum length of 1000, so the prankster has enough room to answer but cannot ramble forever.

And if memory is wired up, working notes and session summaries feed back in here — but we'll get into that separately. The point is Studio lets you audition any combination of settings before you commit anything to the registry file.
